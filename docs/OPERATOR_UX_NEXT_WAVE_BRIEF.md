# Service Butler UX Next Wave Brief

Status: Draft for execution
Audience: Product, Design, Frontend, QA, RevOps
Scope: Product language and UX direction only

## Why the product still feels commodity

- It reads like a generic CRM in several places, with too many equal-weight cards and actions.
- The scanner and dashboard still compete for attention instead of behaving like a single command center.
- Trust is present, but not always felt, because provenance and evidence are not prominent enough.
- The app sometimes looks like a grid of data boxes instead of a workflow for operators.
- The current hierarchy does not always make the next best action obvious in under five seconds.

## What makes it one-of-a-kind

- It can behave like a demand command center, not just a lead list.
- It can show why a lead exists, not just that it exists.
- It can move from signal to verified lead to outreach to booked job in one operator flow.
- It can surface local-service demand with operator-relevant context, not generic marketing noise.
- It can make booked jobs the core narrative, which is more valuable than lead volume.

## Next UX wave brief

The next wave should make three things unmistakable:

1. Scanner is the command center.
2. Verified leads are the primary queue.
3. Trust and next action are visible without drilling into detail.

### 1. Command-center hierarchy

- Make scanner the first and strongest operational surface.
- Show market, urgency, job type, and primary CTA above the fold.
- Use one dominant action per screen, not multiple competing clusters.
- Replace box-heavy layouts with flowing sections, summary strips, and evidence rails.

### 2. Trust and provenance presentation

- Every lead and opportunity should show source, freshness, verification state, and evidence.
- The UI should explain why a lead is worth attention in one sentence.
- Verified status should always be backed by visible proof.
- Uncertainty should be explicit, not hidden behind neutral styling.

### 3. Operator-first differentiation

- The product should feel built for dispatchers and owners, not generic SaaS users.
- Copy should use operator language like verified lead, next action, booked job, and source evidence.
- The workflow should reward speed to action, not browsing.
- Mobile should feel like a real work surface, not a shrunken desktop grid.

## Top 5 recommendations

1. Make scanner the command center, with a clear market bar, scan intent, and one dominant CTA.
2. Turn the lead inbox into a verified queue with visible proof, urgency, and contactability.
3. Surface provenance everywhere a user is asked to trust a lead.
4. Reduce boxy grid behavior by using sectioned layouts, summary strips, and fewer nested cards.
5. Design every screen around one obvious next action, especially on mobile.

## Implementation target

If a screen does not answer these questions quickly, it is not ready:

- What is happening?
- Why does it matter?
- Can I trust it?
- What should I do now?
